## Repository Overview

OS-APOW is a **GitHub Actions-based AI orchestration system** that transforms GitHub Issues into "Execution Orders" autonomously fulfilled by specialized AI agents. It represents a paradigm shift from interactive AI coding to headless agentic orchestration.

### Core Concept

The system implements a **Four-Pillar Architecture**:

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   THE EAR    │────▶│  THE STATE   │────▶│  THE BRAIN   │
│   (Notifier) │     │  (Work Queue)│     │  (Sentinel)  │
│   FastAPI    │     │ GitHub Issues│     │   Polling    │
└──────────────┘     └──────────────┘     └──────────────┘
                                                │
                                                ▼
                                       ┌──────────────┐
                                       │  THE HANDS   │
                                       │   (Worker)   │
                                       │  DevContainer│
                                       └──────────────┘
```

**The Ear (Notifier):** FastAPI webhook receiver for GitHub events  
**The State (Work Queue):** GitHub Issues + Labels as a state machine  
**The Brain (Sentinel):** Polling orchestrator managing worker lifecycles  
**The Hands (Worker):** Isolated DevContainer executing AI agents

## Key Patterns & Conventions

### Four-Pillar Architecture

The system is strictly decoupled across four conceptual pillars:

| Pillar | Technology | Role |
|--------|------------|------|
| **The Ear** | FastAPI, Pydantic, HMAC | Webhook receiver for GitHub events |
| **The State** | GitHub Issues, Labels | Persistence layer and state machine |
| **The Brain** | Python async, HTTPX | Sentinel orchestrator managing workers |
| **The Hands** | opencode CLI, DevContainer | Isolated execution environment |

### Task State Machine

Tasks progress through states represented by GitHub labels:

```
[agent:queued] → [agent:in-progress] → [agent:success]
                        ↓
                 [agent:error]
                        ↓
               [agent:infra-failure]
```

| Label | Meaning |
|-------|---------|
| `agent:queued` | Task validated, awaiting Sentinel pickup |
| `agent:in-progress` | Sentinel has claimed and is executing |
| `agent:reconciling` | Stale task being recovered (no updates for 15+ min) |
| `agent:success` | Terminal success state |
| `agent:error` | Technical/execution failure |
| `agent:infra-failure` | Infrastructure failure (timeout, OOM, container crash) |
| `agent:stalled-budget` | Budget/token limit exceeded |

### Coding Conventions

- **Keep changes minimal and targeted**
- **Do not hardcode secrets/tokens**
- **Pin action versions by SHA in workflow files**
- **Never add duplicate top-level `name:`, `on:`, or `jobs:` keys in workflow YAML**
- **Repository labels defined in `.github/.labels.json`** - use `scripts/import-labels.ps1` to sync
- **Orchestrator agent delegates to specialists via `task` tool** - never writes code directly
- **Preserve `__EVENT_DATA__` placeholder in `orchestrator-agent-prompt.md`**
- **Dockerfile lives at `.github/.devcontainer/Dockerfile`**
- **Consumer devcontainer uses `"image:"` - no local build**

### Python Code Style

- **Line length:** 100 characters
- **Formatter:** Ruff (replaces Black)
- **Linter:** Ruff (replaces flake8, isort)
- **Type checking:** MyPy strict mode
- **Async:** Use `async/await` throughout
- **Validation:** Pydantic for all data models

### Git Conventions

- **Branch naming:** `feature/`, `fix/`, `refactor/`, `docs/`
- **Commit messages:** Follow conventional commits (feat, fix, docs, style, refactor, test, chore)
- **PR workflow:** Feature branch → main

### Error Handling Patterns

# Use Pydantic for validation
from pydantic import BaseModel, Field

class WorkItem(BaseModel):
    issue_number: int = Field(..., gt=0)
    title: str = Field(..., min_length=1, max_length=500)
    status: str = Field(default="agent:queued")

# Use async context managers for resources

    response.raise_for_status()

### Security Considerations

- **Never log or expose `GITHUB_TOKEN`**
- **Use `scrub_secrets()` before posting to GitHub comments**
- **Validate webhook signatures with HMAC SHA256**
- **Worker containers run in isolated Docker networks**
- **Credentials are ephemeral - injected via env vars, never persisted**

### Performance Guidelines

- **Use async/await** - Don't block the event loop
- **Connection pooling** - Reuse HTTPX client
- **Polling interval** - Default 60s, configurable via `POLLING_INTERVAL_SECONDS`
- **Worker limits** - 2 CPUs, 4GB RAM per container

## Troubleshooting

### Common Issues

#### 1. Missing Environment Variables

**Error:** `ValidationError: GITHUB_TOKEN is required`

**Solution:**
```bash
cp .env.example .env
# Edit .env with your values
```

#### 2. uv Not Found

**Error:** `uv: command not found`

**Solution:**
```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
source ~/.bashrc  # or ~/.zshrc
```

#### 3. Docker Permission Denied

**Error:** `permission denied while trying to connect to the Docker daemon`

**Solution:**
```bash
sudo usermod -aG docker $USER
# Log out and log back in
```

#### 4. Prebuilt Devcontainer Image Not Found

**Error:** `manifest unknown` when pulling devcontainer image

**Solution:** This is expected on fresh clones. The `validate.yml` workflow handles this by building a fallback image locally.

#### 5. Tests Failing Due to Missing Dependencies

**Error:** `ModuleNotFoundError: No module named 'pytest'`

**Solution:**
```bash
uv sync --dev
source .venv/bin/activate
```

#### 6. PowerShell Script Execution Policy

**Error:** `cannot be loaded because running scripts is disabled on this system`

**Solution:**
```powershell
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
```
